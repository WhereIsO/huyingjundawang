# 前后端开发流程与实现细节

本文档记录本项目从需求拆解到前后端集成的真实开发流程，供实验报告、答辩 PPT、UML 建模和 Claude Code 二次加工使用。

## 1. 项目目标

课程要求实现一个 Windows 平台数据备份软件。基本要求是数据备份和数据还原；本组选做并实现的扩展功能包括：

- 文件类型支持：普通文件、目录、空目录、符号链接、硬链接、Junction；未知 ReparsePoint 尽力恢复。
- 元数据支持：大小、属性、创建/访问/修改时间、Owner SID、ACL 摘要。
- 自定义备份：路径、类型、名称、时间、尺寸、用户 6 类筛选。
- 打包解包：单文件 `.pbackup` 归档格式。
- 压缩解压：自研哈夫曼压缩。
- 加密解密：Windows CNG AES-256-GCM，PBKDF2-SHA256。
- 图形界面：Qt 5 Widgets。
- 自动化构建与测试：CMake + Ninja + GoogleTest / CTest。

不做项：

- 定时备份。
- 实时备份。
- 网络备份。

这些不做项应在报告中明确说明为范围取舍，不应写成“视进度”。

## 2. 技术栈与环境

| 项 | 当前选择 |
|---|---|
| 操作系统 | Windows 11 x64 |
| 开发语言 | C++17 |
| 编译器 | MSVC 19.44 / Visual Studio 2022 Community |
| GUI | Qt 5.12+ Widgets + Concurrent；运行版已在 `bin` 自带 Qt/VC 运行库 |
| 构建 | CMake + Ninja |
| 测试 | GoogleTest + CTest |
| 加密 API | Windows CNG (`bcrypt`) |
| 压缩 | 自研 Huffman |

重要环境约束：

- 构建时 Qt 的 ABI 必须与编译器匹配，MSVC 版 Qt 需要用 MSVC 构建。
- 源文件是 UTF-8，MSVC 必须加 `/utf-8`。
- 项目路径含中文时，Qt AUTOMOC 可能出问题，因此 `build_msvc.bat` 默认镜像到 `%TEMP%\BackupTool_src` 后构建。

## 3. 总体开发流程

### 3.1 需求拆解

第一步不是直接写代码，而是把评分项拆成两类：

- 必做基础功能：备份、还原。
- 必做扩展功能：文件类型、元数据、6 类筛选、打包解包、压缩解压、加密解密、GUI。

这一步同时确定后端禁止使用 Python 等脚本语言，压缩禁止第三方库，加密必须使用成熟密码学 API。

### 3.2 契约优先

前端和后端通过 `BackendAdapter.h` 对接。该文件是 GUI 层唯一认可的后端接口，核心结构包括：

- `BackupRequest`
- `RestoreRequest`
- `FilterSpec`
- `Progress`
- `BackendAdapter`
- `createBackend()`

契约设计的目的：

- GUI 不直接依赖 core，方便先做界面。
- 后端可独立测试，不被 Qt 绑死。
- 后端完成后通过 `RealBackend` 做类型转换和线程桥接。

### 3.3 前端先行与 MockBackend

前端先实现：

- `MainWindow`
- `BackupTab`
- `FilterTab`
- `RestoreTab`
- `LogPanel`
- `PathPicker`
- `Theme`
- `BackendAdapter` + `MockBackend`

`MockBackend` 的作用是让 GUI 在真实后端未完成时也能运行、展示进度条和日志，降低并行开发阻塞。

### 3.4 后端核心实现

真实后端放在：

```text
gui/BackupTool/src/core
```

核心层不 include Qt 头文件，只使用标准库、Win32 API 和 Windows CNG。

模块划分：

| 模块 | 文件 | 职责 |
|---|---|---|
| 公共类型 | `types.*` | 错误码、日志级别、进度、选项、文件类型 |
| 编码 | `encoding.*` | UTF-8、宽字符、路径转换、长路径前缀 |
| 校验 | `checksum.*` | CRC32、SHA-256 |
| 二进制 IO | `binary_io.*` | 小端 POD、TLV、字符串序列化 |
| 筛选 | `filter.*` | 路径/类型/名称/时间/尺寸/用户 |
| 压缩 | `huffman.*` | 自研哈夫曼压缩与解压 |
| 加密 | `crypto.*` | AES-256-GCM、PBKDF2-SHA256、随机盐 |
| 元数据 | `metadata.*` | 属性、时间、Owner SID、ACL 摘要、Junction 创建 |
| 扫描 | `scanner.*` | 目录树遍历、文件类型识别、硬链接识别 |
| 归档 | `archive.*` | `.pbackup` 写入、读取、校验 |
| 任务 | `backup_task.*` | 备份/还原流程调度 |

### 3.5 RealBackend 桥接

`RealBackend` 位于：

```text
gui/BackupTool/src/RealBackend.h
gui/BackupTool/src/RealBackend.cpp
```

它继承 `pbackup::ui::BackendAdapter`，负责：

1. 接收 GUI 的 `BackupRequest` / `RestoreRequest`。
2. 把 `QString` 路径转换成 `std::filesystem::path`。
3. 把 `FilterSpec` 转成 core 的 `FilterRules`。
4. 用 `QtConcurrent::run` 启动后台线程。
5. 把 core 的进度回调转成 Qt `progress` 信号。
6. 把 core 的日志回调转成 Qt `log` 信号。
7. 处理取消、异常、完成信号。

路径转换原则：

- GUI 层用 `QString`。
- core 层用 `std::filesystem::path`。
- 中文路径通过宽字符转换，避免 GBK/UTF-8 混乱。

线程模型：

- GUI 主线程只负责界面响应。
- 备份/还原在 QtConcurrent 工作线程执行。
- 进度通过 Qt queued connection 回到主线程。
- `m_cancel` 是 `std::atomic_bool`，core 轮询后尽快中断。

### 3.6 工厂开关

`createBackend()` 位于 `BackendAdapter.cpp`。当前逻辑：

```cpp
if (qgetenv("BACKUP_BACKEND_MODE").toLower() == "real") {
    return createRealBackend(parent);
}
return std::make_unique<MockBackend>(parent);
```

含义：

- 默认 MockBackend，适合纯界面演示。
- 设置 `BACKUP_BACKEND_MODE=real` 后使用真实后端。

## 4. `.pbackup` 格式实现

格式文档见：

```text
gui/BackupTool/docs/format.md
gui/BackupTool/src/core/docs/format.md
```

总体结构：

```text
FileHeader    固定 64 字节
GlobalConfig  TLV
IndexEntry[]  TLV
Payload       文件数据，可压缩，可整体加密
Trailer       PayloadCRC32 + PackageSHA256
```

关键校验：

- HeaderCRC32：校验头部前 60 字节。
- EntryCRC32：校验每个索引条目。
- PayloadCRC32：校验实际存储 payload。
- PackageSHA256：校验整个包。
- AES-GCM Tag：校验密码和密文完整性。

压缩顺序：

```text
单文件数据 -> 哈夫曼压缩 -> 写入明文 payload
```

加密顺序：

```text
所有明文 payload -> AES-256-GCM 整体加密 -> IV + Tag + CipherText
```

## 5. 文件类型与元数据实现

### 5.1 普通文件与目录

普通文件读取内容写入 payload。目录和空目录只写索引，不写 payload。

### 5.2 硬链接

扫描时通过 `GetFileInformationByHandle` 获取 `nFileIndexHigh/Low`。同一卷内 file index 相同的文件视为硬链接：

- 第一个文件按普通文件写入数据。
- 后续同 file index 条目标记为 `Hardlink`，只记录目标相对路径。
- 还原时调用 `CreateHardLinkW`。
- 如果创建失败，尝试复制目标内容并记录 warning。

### 5.3 符号链接

扫描时识别 symlink 并记录目标路径。还原时调用 `CreateSymbolicLinkW`。Windows 创建 symlink 可能需要开发者模式或管理员权限，因此测试中如果无权限会跳过。

### 5.4 Junction

Junction 属于目录 reparse point。当前实现已支持真实恢复：

- 扫描时用 reparse 数据读取目标路径。
- 备份时索引 entry type 为 `Junction`，payload 不写文件数据。
- 还原时创建目录并调用 `FSCTL_SET_REPARSE_POINT` 设置 mount point。
- 自动化测试 `ScannerTest.DetectsJunctionWhenSupported` 和 `BackupRestoreTest.JunctionRoundTripWhenSupported` 已通过。

开发中修复过一个关键问题：`std::filesystem::relative()` 在遇到 Junction 时会跟随目标路径，导致相对路径变成 `..\target`。当前改用 `lexically_relative()` 做纯词法相对路径，避免跟随 reparse point。

### 5.5 未知 ReparsePoint

未知 reparse point 不强行恢复具体语义，避免误操作。当前策略：

- 识别并记录为 `ReparsePoint`。
- 还原时按目录尽力恢复。
- 记录 warning。

## 6. 筛选实现

`FilterSpec` 到 `FilterRules` 的映射由 `RealBackend` 完成。

| GUI 字段 | core 字段 | 说明 |
|---|---|---|
| includePath | includePath | 路径包含 |
| excludePath | excludePath | 路径排除 |
| nameGlob | nameGlob | 通配符 |
| typeFilter | typeFilter | file/dir/symlink/hardlink/junction/reparse |
| sizeMin/sizeMax | sizeMin/sizeMax | 字节 |
| mtimeAfter/mtimeBefore | mtimeAfterNs/mtimeBeforeNs | 日期转 Unix ns |
| ownerSid | ownerSid | Windows 所有者 SID |

规则：

- 每类为空表示不过滤。
- 多个类别同时非空时按 AND 生效。
- includePath 支持多个包含子串。
- excludePath 命中即排除。

## 7. 加密实现

加密模块使用 Windows CNG，不手写密码学算法。

实现要点：

- 算法：AES-256-GCM。
- KDF：PBKDF2-SHA256。
- Salt：8 字节随机盐，写入 header。
- IV：12 字节。
- Tag：16 字节。
- 默认 KDF 迭代：200000。
- 错误密码会导致 AES-GCM 认证失败并抛出 `BackupError`。

说明：课程提到 OpenSSL / Crypto API 均可。本项目选择 Windows CNG，减少 OpenSSL 部署复杂度，同时符合“加密允许使用第三方/系统密码学 API，禁止手搓算法”的要求。

## 8. 压缩实现

压缩模块为自研哈夫曼，不使用第三方库。

实现要点：

- 每个普通文件独立压缩。
- 空文件、单字符重复文件、普通文本、二进制数据均有测试。
- 解压时校验头部和码流完整性。

## 9. 构建系统实现

`gui/BackupTool/CMakeLists.txt` 当前负责：

- 构建 `BackupTool.exe`。
- 添加 core 源文件。
- 添加 `RealBackend`。
- 链接 Qt5 Widgets / Concurrent。
- 链接 Windows `bcrypt` 和 `Advapi32`。
- 优先使用本机 GoogleTest；仅在 `BACKUP_ALLOW_GTEST_DOWNLOAD=ON` 时允许 CMake 下载 GoogleTest。
- 创建 `backup_tests`、`real_backend_smoke`、`gui_signal_smoke`。

`build_msvc.bat` 负责：

- 用 `robocopy /MIR` 镜像源码到 `%TEMP%\BackupTool_src`，也可用 `ASCII_SRC` 覆盖。
- 调用 VS 2022 `vcvars64.bat`。
- 配置 CMake。
- 构建。
- 按 `RUN_TESTS` 设置决定是否运行 CTest。

测试覆盖类别超过 4 种：

- 单元测试。
- 集成测试。
- GUI 信号 smoke test。
- 异常/健壮性测试。
- 文件系统特殊类型测试。
- 加密错误密码测试。

历史验证结果：

```text
100% tests passed, 0 tests failed out of 43
2 项 symlink 权限相关测试跳过
```

跳过原因：Windows 当前环境不允许创建符号链接。不是代码失败；如果启用开发者模式或管理员权限，测试可尝试执行。当前源码新增了第 44 个路径穿越防护用例，需在具备 Qt/MSVC/GTest 的构建环境中重新跑一次 CTest 更新截图和报告数据。

## 12. 适合写入报告/PPT的开发流程总结

1. 根据评分表拆解基础功能和扩展功能。
2. 先确定 `BackendAdapter` 前后端契约，避免接口漂移。
3. 前端先用 MockBackend 完成 GUI、日志、进度、参数收集。
4. 后端在 `src/core` 中实现纯 C++ 核心，保持可单测。
5. 用 `RealBackend` 将 Qt 类型、线程模型、信号槽和 core 回调连接起来。
6. 通过 `BACKUP_BACKEND_MODE=real` 做真实后端切换。
7. 用 CMake + Ninja + MSVC 完成自动化构建。
8. 用 GoogleTest / CTest 覆盖 44 项源码测试用例。
9. 修复中文路径、MSVC UTF-8、Qt ABI、Junction 相对路径、还原路径穿越等真实工程问题。
10. 形成可运行软件、格式文档、README、使用说明、实验报告。
