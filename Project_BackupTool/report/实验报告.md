本课程的教学目标是为学生提供一个软件实验综合的基础平台。以计算机科学与技术专业本科课程《程序设计（C／C++）》《数据结构与算法设计》《软件工程》等主干课程所涉及的重要知识为核心，依托具体的工程项目，引导学生运用这些知识解决问题，使学生能够解决具体软件系统的复杂工程问题，达到选择适宜、更高级结构（存储结构）与相应算法的设计目标，加深学生对实际问题描述、抽象和运用多门课程知识进行综合处理的能力，培养学生的计算思维、抽象算法、软件设计与程序设计、软件实现调试、团队协作、复杂软件系统的认知、方案设计和评估分析的软件工程实践能力。

## 需求分析说明书（10 分）

### 1. 引言概述

#### 1.1 引言

本文档面向"数据备份软件"这一软件产品，描述其功能、外部接口与运行约束，供后续设计、编码与测试阶段参照。文档读者包括项目组全体成员、指导教师及验收方。

#### 1.2 综合描述

**1.2.1 产品现状**

本产品是一个**独立**的桌面级数据备份工具，面向 Windows 平台，不依赖任何在线服务，也不作为其他产品的组件运行。它是一个自含型的新产品，通过图形界面向用户提供备份、还原、筛选、压缩、加密等能力，其数据落盘为自定义的 `.pbackup` 归档包。

外部实体包括个人用户、Windows 文件系统、Windows CNG / Crypto API。顶层用例图与结构图建议在后续 StarUML 建模阶段导出到 `docs/uml/`。

**1.2.2 产品的功能**

本产品应具备的主要功能可归纳为：

- **备份**：遍历源目录树，将其中文件（含子目录结构）打包写入单一 `.pbackup` 备份包；
- **还原**：读取 `.pbackup` 备份包，将其中文件按原目录结构还原到指定位置；
- **筛选**：备份前按路径、类型、名称、时间、尺寸、用户六类条件过滤待备份文件；
- **压缩**：对文件数据做哈夫曼压缩以节省存储空间；
- **加密**：使用用户口令对备份包做 AES-256 加密，还原时校验口令；
- **进度与日志**：实时展示扫描/过滤/写入/校验各阶段进度，并输出彩色分级日志；
- **特殊文件与元数据保留**：识别并记录符号链接、硬链接、junction，保留文件属性、时间戳、所有者与 ACL。

**1.2.3 用户特点**

本产品的直接使用者为**个人 PC 用户**（学生、开发者、普通办公用户），假定其具备基本的文件系统操作常识，但不假定其了解备份包格式、压缩/加密算法等技术细节。因此界面设计遵循"零技术暴露、全流程自动"的原则：用户只需选择源目录、备份包位置并勾选是否压缩/加密，其余由软件自动完成，不向用户暴露 max_tokens、协议、UA 等技术字段。此外，界面采用大字号、大按钮、高对比度配色，以照顾视力不佳的用户。

#### 1.3 运行环境

**1.3.1 软件环境**

- 操作系统：Windows 10 / 11（x64）
- 运行时依赖：Qt 5.15 运行库（Qt5Widgets.dll、Qt5Core.dll、Qt5Concurrent.dll 及平台插件 platforms/qwindows.dll）；本机 Qt 位于 `G:\Anaconda3\Library`

**1.3.2 硬件环境**

- 内存：≥ 2 GB
- 硬盘：≥ 100 MB 可用空间（另需与备份数据等量的目标空间）
- 网络：无需（本地备份）

### 3. 外部接口需求

#### 3.1 用户界面

采用 Qt 5 Widgets 桌面图形界面，主窗口为"顶部标签页 + 底部进度与日志"布局：

- **顶部标签页**：① 备份、② 筛选、③ 还原三页；
- **底部**：状态行 + 进度条 + 可拉伸的彩色日志面板；
- **控件规范**：主按钮 240×64、输入框最小高 40，字体 Microsoft YaHei UI 14pt（可由环境变量 `BACKUP_FONT_SIZE` 覆盖，范围 10–28），主色 #1565C0、危险色 #C62828，高对比度；
- **信息提示**：缺参数、口令为空、任务失败等均以模态对话框或彩色日志显式提示。

#### 3.2 硬件接口

无特殊硬件接口，读写本地磁盘经由操作系统文件 API。

#### 3.3 软件接口

- **操作系统**：Windows 文件系统 API（读写文件、遍历目录、读取属性/时间戳/ACL、识别重解析点）；
- **加密库**：Windows CNG / Crypto API（AES-256），此为课程唯一允许调用第三方库的功能点，其余算法（哈夫曼）自研；
- **GUI 与后端接口**：前端仅依赖抽象基类 `BackendAdapter`（定义于 `BackendAdapter.h`），不直接依赖后端实现。契约以信号 `progress / log / finished / failed` 异步回调，以 `startBackup / startRestore / cancel / isRunning` 为入口。当前同时提供 `MockBackend` 与 `RealBackend`，设 `BACKUP_BACKEND_MODE=real` 时使用真实后端，不设置时回退 Mock 后端。

### 4. 其他非功能性需求

#### 4.1 性能需求

- 备份/还原在工作线程执行（`QtConcurrent::run`），界面主线程不阻塞；
- 日志面板设最大块数 5000 行，自动滚动，避免长任务下内存无界增长；
- 目标：GB 级目录备份过程中界面保持可交互、进度条平滑更新。

#### 4.2 安全性需求

- 加密口令仅在内存中传递，进入后端后按约定清零，不写入日志、不回显；
- 备份包携带完整性校验字段，还原前先校验，防止损坏包被误还原；
- 采用系统级 AES-256，不自行实现密码学原语。

#### 4.3 软件质量属性

- **可用性**：大字号、大按钮、全自动流程，降低使用门槛；
- **可移植性**：Qt + CMake，切换 Qt 6 仅需改一行版本号；跨中文/英文路径需注意工具链编码（见"改进建议"）；
- **可维护性**：前后端以抽象接口解耦，GUI 与核心逻辑分离，核心可被单元测试直接调用。

### 2. 概念设计

#### 2.1 系统结构设计

本软件采用**分层 + 前后端解耦**结构：

```
┌─────────────────────────────────────────┐
│          表示层（Qt GUI）                    │
│  MainWindow / BackupTab / RestoreTab /   │
│  FilterTab / LogPanel / PathPicker       │
└───────────────┬─────────────────────────┘
                │ 抽象契约 BackendAdapter
                │ (progress/log/finished/failed 信号)
┌───────────────┴─────────────────────────┐
│  适配层  MockBackend  |  RealBackend      │
└───────────────┬─────────────────────────┘
                │
┌───────────────┴─────────────────────────┐
│        核心层（纯 C++ 后端，不依赖 Qt）        │
│  BackupTask / RestoreTask / Scanner      │
│  FilterRules / Metadata                  │
│  Archive read/write                      │
│  Huffman / Windows CNG Crypto            │
│  CRC32 / SHA-256                         │
└──────────────────────────────────────────┘
```

**2.1.1 总体系统结构**

**2.1.1.1 结构图** — 当前报告给出文字结构图；正式提交前建议用 StarUML 导出顶层用例图、包图、类图和顺序图到 `docs/uml/`。

**2.1.1.2 结构图描述** — 表示层通过抽象接口向下调用，核心层不反向依赖 GUI；默认可用 Mock 独立运行与演示，设置 `BACKUP_BACKEND_MODE=real` 后使用真实后端完成备份与还原。

**2.1.2 子系统一：前端 GUI 结构**

- `MainWindow`：装配三标签页 + 进度条 + 日志面板，持有 `BackendAdapter`，把界面事件转为后端调用、把后端信号转为界面更新；
- `BackupTab / RestoreTab / FilterTab`：分别收集 `BackupRequest / RestoreRequest / FilterSpec`；
- `LogPanel`：`QPlainTextEdit` 派生，分级彩色日志，带时间戳与自动滚动；
- `PathPicker`：标签 + 输入框 + 浏览按钮的复用控件；
- `Theme`：集中管理字号、控件尺寸、配色（大字号策略）。

### 八、实验结论：

本组完成了数据备份软件的 Qt 图形界面、真实 C++ 后端核心与前后端桥接。前端基于 Qt 5 + CMake，采用前后端解耦架构：GUI 仅依赖抽象基类 `BackendAdapter`，默认可用 `MockBackend` 独立演示，设置环境变量 `BACKUP_BACKEND_MODE=real` 后切换为 `RealBackend`，调用纯 C++ `src/core` 执行真实备份与还原。实测在 MSVC 19.44 + Qt 5.15.2 下全量构建通过，生成 `G:\_build_backup_tool_gui_ascii\BackupTool.exe`；最近一次 CTest 运行共 43 项，全部通过，0 失败。界面满足大字号、大按钮、高对比度、全流程自动的设计目标，功能覆盖备份、还原、六类筛选、压缩、加密、实时进度与彩色日志等。

### 九、总结及文档心得：

- **前后端契约先行**：先定义 `BackendAdapter` 抽象接口与数据结构，前端得以在后端未就绪时用 Mock 独立推进；后端完成后只需在 `createBackend()` 中按环境变量返回 `RealBackend`，降低了团队并行开发的耦合风险。
- **工程环境是隐性成本**：本项目在环境上踩了三个真实坑——中文路径导致 MinGW 报 Illegal byte sequence、Anaconda 的 Qt 为 MSVC 构建导致与 MinGW ABI 不匹配链接失败、MSVC 默认按 GBK 读 UTF-8 源文件导致 C2001/C1057。解决办法分别是：改用 ASCII 短路径构建、改用 VS 2022 的 MSVC 工具链、为 MSVC 显式添加 `/utf-8`。这说明"能编译"本身也是软件质量的一部分，需在文档中固化环境约束。
- **文档规范性**：按需求分析→系统设计→测试报告的顺序组织，用例、类描述、顺序图、状态图一一对应，保证了一致性与可读性。

### 十、对本实验过程及方法手段的改进建议：

1. **统一构建环境**：建议课程明确推荐工具链组合（如 VS 2022 MSVC + 官方 Qt，或 MinGW + MinGW 版 Qt），避免混用导致的 ABI 冲突；并在项目 CMake 中默认加入 `/utf-8`（MSVC）与短路径构建建议。
2. **CI 早接入**：尽早接入持续集成，把"全量编译通过"作为最低门槛，避免集成阶段才暴露环境问题。
3. **Mock 契约共享**：前后端在需求阶段即共同评审 `BackendAdapter` 契约，减少后期返工。
